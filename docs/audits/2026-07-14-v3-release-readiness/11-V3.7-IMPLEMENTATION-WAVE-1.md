# V3.7 implementation wave 1 — isolation and daemon identity

Date: 2026-07-14
Release: `3.7.0` Memory Integrity
Status: implemented in the working tree; independent and built-artifact verification remain open

## Decision

Three designs were considered.

| Design | Cost and reversibility | Failure left behind | Decision |
|---|---|---|---|
| Add a service marker to `/health` | Low cost; easy to revert | PID reuse, stale files, foreign default ports, global stop/restart, and split namespaces remain | Rejected |
| Namespace plus process descriptor | Moderate cost; additive one-release compatibility | Requires client/server contract migration and broader runtime verification | Selected |
| Replace TCP discovery with Unix sockets or named pipes | High cost; OS-specific migration and client breakage | Strongest transport isolation, but incompatible with the V3.7 client and remote-mode schedule | Deferred research |

The selected pattern is a namespace-aware process descriptor. PID and port are liveness hints; they are never ownership credentials.

## Implemented contract

- One canonical environment precedence is used by the descriptor: `SLM_DATA_DIR`, then `SL_MEMORY_PATH`, then `SLM_HOME`, then the default.
- Each data root has a domain-separated namespace hash.
- Each daemon process has a random instance ID and private capability.
- `daemon.json` is written atomically with mode `0600` and contains service, protocol, namespace, instance, capability fingerprint, owner, PID, process creation time, actual port, state, version, and timestamps.
- `/health` retains compatibility fields and adds the identity contract.
- Clients require matching service, protocol, namespace, instance, capability fingerprint, owner, version, PID, process creation time, and port.
- Redirected health responses, foreign health responses, malformed descriptors, stale PIDs, PID reuse, and fixed-port fallback are rejected.
- Custom daemon ports no longer probe legacy port `8767`; the inbound compatibility redirect targets the actual runtime port.
- Client requests carry the private capability and target instance after health attestation.
- `/stop` requires both the capability and current target instance.
- `stop_daemon` and `cmd_restart` no longer scan or kill machine-wide SLM processes by command name.

## Test isolation implemented

- Pytest establishes a private HOME, data aliases, XDG roots, daemon port, and capability before collection imports.
- Direct file opens under the captured real SLM data root are denied by a process audit hook.
- Socket connections to live public ports `8765` and `8767` are denied.
- Child Python processes inherit the isolated namespace.
- Inherited hook, mesh, host, peer, and secret variables are cleared.
- Daemon auto-spawn is denied in tests unless an explicit fixture opts in.
- The global `pkill -f` cleanup path is removed.
- The old async-remember E2E cannot probe a live daemon during collection. It now requires a fixture-owned `SLM_TEST_DAEMON_URL` and isolated database paths.

## Additional defect fixed

`slm doctor` treated only `ImportError` as dependency failure. A discovered module can raise another exception during import when its transitive installation is incompatible. Doctor now records that dependency as unavailable and continues instead of crashing.

## Not yet verified or closed

This wave does not close `SLM-P0-003` or `SLM-P0-005` as verified.

- A built wheel/sdist has not yet completed the two-HOME, foreign-port, service-manager, upgrade, and Windows matrix.
- The fixture-owned real daemon E2E and owned process-group cleanup fixture are not yet implemented; the legacy live E2E is safely skipped.
- The server still has daemon subsystems that resolve `Path.home()/.superlocalmemory` directly instead of the canonical data root. This is registered as `SLM-P1-016` and blocks custom-root parity.
- The public website/README/wiki/CLI benchmark and product-claim rewrite is inventoried but not implemented in this wave.
- Default ingestion parity, scope-correct retrieval, and the unified untrusted-memory injection renderer remain the next P0 implementation wave.
- Current V3.7 LoCoMo performance remains unknown. No LoCoMo rerun or current score is claimed.
