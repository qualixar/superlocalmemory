# V3.7 implementation wave 2B — installer rebuild and activation boundary

Date: 2026-07-14
Release: `3.7.0` Memory Integrity
Status: implementation complete in the working tree; hosted cross-platform proof and the DMG decision remain open

## Outcome

Option B was approved. The supported npm, clone-based macOS/Linux, and clone-based
Windows paths now install application code into isolated environments. They do not
bootstrap a tool manager, mutate protected Python, configure an IDE, install hooks,
start a daemon, download models, or create a memory data root.

The activation pattern is deliberately two-stage:

1. `npm install -g superlocalmemory`, `uv tool install`, `pipx install`, or a clone
   installer establishes the executable environment only.
2. `slm setup` is the explicit consent boundary for profile selection, models,
   external IDE/plugin integration, hooks, and autostart.

npm lifecycle scripts do not prompt. npm does not provide a reliable interactive TTY
contract across terminals, CI, package managers, and global installs. The guided product
experience therefore begins in the explicit `slm setup` command.

This wave is not a V3.7 release approval. The repository is not a frozen release
candidate, the hosted Windows/macOS/Linux matrix has not run, the DMG channel is not
resolved, and other defect-ledger workstreams remain open.

## Supported channel contract

| Channel | Executable ownership | Setup behavior | Upgrade/uninstall behavior | Current proof |
|---|---|---|---|---|
| npm | Package-owned `.slm-venv`; wrapper invokes its exact interpreter | No implicit setup or external edits | npm owns code removal; durable data is never inspected | Packed tarball lifecycle passed on macOS |

| Repository clone, Windows | `scripts/install.ps1` delegates to an existing uv or pipx | No execution-policy, registry, PATH, or config mutation | automatic upgrade/uninstall detect ownership and fail closed on dual ownership | PowerShell parser/static contracts passed; runtime matrix pending hosted Windows CI |
| DMG | Obsolete V2 builder, not a valid V3.7 channel | Unsafe and false contract | Not release-linked | Registered as `SLM-P1-019`; decision required |

## Implemented safety boundaries

The npm installer validates Python 3.11–3.14, rejects a symlinked private-runtime
directory, installs the current npm artifact into that private environment, and checks
that npm and Python artifact versions agree. Its wrapper strips `PYTHONPATH` and invokes
only the package-owned interpreter.

The Unix and Windows clone installers accept a package, wheel, or repository source and
delegate only to an already-installed uv or pipx. They contain no remote-script pipe,
system pip, `sudo`, protected-Python bypass, shell-profile edit, service install, data-root
mutation, or implicit manager bootstrap. Ambiguous dual ownership fails closed.

Pre-uninstall is code-only. It neither reads memory locations nor prints destructive
data-deletion commands. Public install guidance now distinguishes the npm CLI, isolated
Python CLI, Python SDK virtual environment, and repository clone paths.

CLI metadata inspection is activation-free. Empty invocation, global help, command help,
and version output create no configuration, marker, database, daemon, hook, model, or data
directory.

## Hosted installer matrix

`.github/workflows/installer-matrix.yml` adds least-privilege artifact execution on
Ubuntu 22.04, macOS 14, and Windows latest. It builds the candidate wheel and npm tarball,
runs installer contracts, installs the artifacts, executes activation-free version probes,
asserts that no data root exists, and uninstalls the code. Actions are pinned to immutable
commit SHAs.

The workflow exists and parses locally. It has not run on GitHub in this working tree, so
its platform rows are proof-pending rather than verified.

The local package artifacts still identify as `3.6.23`. That is intentional before the
release candidate is frozen: synchronizing every package/version surface to `3.7.0` is
part of the release-provenance gate, not an ad hoc installer edit. These local artifact
passes prove behavior of the current tree; they are not registry or tag evidence.

## DMG decision

The current DMG scripts cannot ship. Two valid choices remain:

| Choice | Cost and consequence |
|---|---|
| Remove the DMG from V3.7 supported surfaces | Low cost, immediately truthful, fully reversible in a later release. npm, uv, pipx, and clone installers remain supported. |
| Rebuild a signed and notarized V3.7 DMG from frozen candidate artifacts | High cost. Requires Apple Developer signing/notarization credentials, a release-linked wheel, Gatekeeper tests, upgrade/uninstall tests, and explicit artifact provenance. |

The L99 recommendation is to remove the DMG from V3.7 supported surfaces unless signing
and notarization credentials are available for this release. An unsigned script bundle in
a disk image harms credibility more than omitting the channel. No DMG file or script was
deleted or rewritten in this wave because that product decision requires explicit approval.

## Remaining release boundary

`SLM-P0-009` and `SLM-P1-020` are `proof_pending`, not `verified`. The macOS artifact
lifecycles are green, but Windows and hosted macOS/Linux rows must pass from the frozen
release candidate. `SLM-P1-021` and `SLM-P0-012` remain `fix_in_review` until their complete
row-specific security/platform proof is attached.

No package was published, no deployment was performed, no commit was created, and no
current LoCoMo score is claimed.
