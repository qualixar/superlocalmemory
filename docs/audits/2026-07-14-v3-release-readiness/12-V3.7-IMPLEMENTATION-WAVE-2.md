## Outcome

It also proves two independent installed-wheel daemon namespaces outside the source checkout.

This is not a V3.7 release approval. The defect ledger still contains 36 rows, the working tree is not a frozen release candidate, two shipped framework adapters remain broken, the legacy installer policy requires owner approval, and no current LoCoMo result exists.

## Architecture decision

Three approaches were evaluated.

| Design | Cost, risk, and reversibility | Decision |
|---|---|---|
| Patch each `Path.home()` call independently | Lowest immediate cost, but precedence and conflict behavior would continue to drift between subsystems | Rejected |
| Central resolver with dynamic path objects and fail-closed conflict detection | Moderate migration cost; additive and reversible; one contract is executable across all callers | Selected |
| Move all legacy data automatically into a new root | Simplifies the steady state but creates unacceptable merge, rollback, ownership, and corruption risk | Rejected |

The selected pattern is a canonical namespace boundary. The resolver chooses one root; it does not move data. Two populated roots are an operator-visible conflict, never an automatic merge.

## Canonical contract

The precedence is:

```text
SLM_DATA_DIR
  > SL_MEMORY_PATH
  > SLM_HOME
  > explicit configured root
  > valid legacy configured base_dir
  > HOME/.superlocalmemory
```

An explicit environment selection cannot be overwritten by a persisted `base_dir`. Existing unreadable roots fail closed. State paths that must follow an environment change inside a long-lived process use `DynamicStatePath`; they are not cached at import time.

## Two-HOME installed-artifact proof

The release harness builds a wheel and sdist from a copied source snapshot, installs the candidate wheel into a temporary virtual environment, changes to a directory outside the checkout, and executes Python with isolated-path mode. It rejects imports from the source tree.

The daemon black box creates two distinct `HOME` and `SLM_DATA_DIR` roots and two kernel-assigned ports that are not the legacy public ports. For both namespaces it exercises start, status, restart, and stop; verifies descriptor/health identity, PID and process creation time; verifies root-local configuration, logs, markers and databases; and proves restarting or stopping one daemon does not affect the other. Cleanup targets only fixture-owned process descriptors and descendants. It does not run `pkill` or a global process-name scan.

## Hard decision resolved after this checkpoint

The legacy `scripts/install.sh` and `scripts/install.ps1` are present in the npm payload but are unsafe and internally inconsistent. The shell installer is non-runnable from its shipped location; both conflate product code with durable data, ignore the canonical alias set, and can mutate a protected Python installation. The npm postinstall path separately retains a `--break-system-packages` fallback.

| Option | Trade-off |
|---|---|
| A — one-release non-mutating compatibility tombstones | Lowest risk and cost. Existing paths print migration guidance but never create, copy, delete, initialize, edit configuration, invoke pip, or move data. Remove the tombstones in V3.8. |
| B — rebuild isolated installers now | Preserves clone-based installation, but requires a full macOS/Linux/Windows ownership, upgrade, rollback and uninstall matrix around pipx, uv or a private virtual environment. High cost and schedule risk. |

The owner approved **Option B** after this report was written. The npm, Unix/macOS,
and Windows installer paths were rebuilt around isolated application environments;
the protected-Python fallback was removed. The implementation and proof are recorded
in Wave 2B. The obsolete DMG builder is a distinct artifact decision and remains open.
