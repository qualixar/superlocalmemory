## Decision

## Real-user evidence

All timings below are external wall-clock measurements from CLI invocations, not only internal engine timers.

| Witness | Remember | Recall | Materialization and layer evidence |
|---|---:|---:|---|
| Mode A, clean local profile | 0.10–0.12 s queryable | 1.09 s first; 0.18–0.54 s warm | PyTorch-first Apple Silicon warmup reached truthful `ready`; semantic, BM25, entity graph, temporal, Hopfield, spreading activation, semantic/entity/temporal graph edges |
| Mode B, real Ollama (`nomic-embed-text`, `llama3.2`) | 0.09–0.12 s queryable | 2.09 s first immediate; 1.70 s next distinct; 0.35 s warm | Runtime logged `TemporalValidator initialized (mode=b)`; operations completed; entity, semantic, temporal, supersession, and contradiction edges |
| Mode C, real OpenRouter LLM plus local OpenAI-compatible Ollama embeddings | 0.11 s queryable | 3.02 s full agentic | Real HTTP 200 responses from OpenRouter; four canonical facts; correct owner, deployment ring, and review date; all full retrieval evidence present |
| Mode C truthful `--fast` | 0.10 s queryable | 0.28 s immediate; 1.44 s after full materialization | Remote agentic verification and spreading activation absent; semantic, temporal, Hopfield, and entity-graph evidence retained |

Mode C background canonical enrichment took about 15–18 seconds for one synthetic memory and made multiple provider calls. This is asynchronous and does not delay the queryable receipt, but call count and cost remain an optimization target. It must not be advertised as one-second canonical enrichment.

## Storage and lifecycle evidence

For each Mode A/B/C sandbox:

- `memory.db`, `learning.db`, and `audit_chain.db` returned `ok` from `PRAGMA quick_check`;
- M018 operations moved from `queryable` through `enriching` to `complete` with no error;
- memories retained `observation_date` and, where the content contained a date, `referenced_date`;
- graph storage contained real semantic, entity, and temporal edges after related facts materialized;
- isolated daemons and workers were terminated without touching the live global `3.6.19` daemon.

The resource-lifecycle benchmark recorded separately in `benchmarks/resource_lifecycle/RESULTS.md` remains the fixed-state leak witness. This wave sampled process RSS but did not claim a new long-duration leak proof.

## Truthful latency contract

V3.7 must not publish a universal one-second SLA. The supported statement is:

- remember returns a durable, queryable receipt near 0.1 seconds on the release laptop;
- local warm recall is normally sub-second;
- first/cold Mode B recall can be around two seconds;
- full Mode C performs remote agentic verification and was about three seconds in this witness;
- `--fast` is the explicit latency-bounded path and trades spreading activation plus remote agentic verification for speed while preserving the other configured channels.

These are release-laptop observations, not cross-hardware guarantees.

## Release verdict

The source is materially stronger and the three modes now have real runtime evidence. Stable V3.7 remains **no-go** until the broad suite, rebuilt installed-artifact witnesses, MCP handshake, installer matrix, cross-platform CI, and release-integrity gates close. No additional retrieval feature should enter the candidate before those gates finish.
