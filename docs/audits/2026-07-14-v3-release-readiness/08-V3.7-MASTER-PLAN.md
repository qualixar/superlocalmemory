# SuperLocalMemory V3.7 master program

Status: **implementation authorized; Wave 2 locally verified, release gates remain open**

Decision date: 2026-07-14

Source baseline: `main@631c6af`, package `3.6.23`

Public release target: `3.7.0 — Memory Integrity`

## Decision

SuperLocalMemory will publish one stable release, `3.7.0`, after closing every defect registered in [09-V3.7-DEFECT-LEDGER.yaml](09-V3.7-DEFECT-LEDGER.yaml). Development will be internally staged through alpha, beta, and release-candidate gates. No `3.6.24` package will be promoted as the stable public release.

The public website and campaign are not allowed to retain known false statements while V3.7 is built. Claim containment is an immediate documentation correction, not a product release.

## Meaning of “all gaps resolved”

An unbounded promise to fix every possible future weakness is not executable. V3.7 instead closes every currently registered P0, P1, and P2 row using one of four explicit closure modes:

| Closure mode | Meaning | Valid proof |
|---|---|---|
| `fix` | The defective contract is implemented correctly. | Executable regression plus required integration/fault tests. |
| `remove` | Dead or unsafe behavior and its public contract are removed. | Absence test, generated-surface scan, migration/compatibility evidence. |
| `narrow` | Product scope is made truthful and the supported subset is hardened. | Contract test for supported behavior and zero stale broad claims. |
| `prove` | An existing capability is retained only after release-linked evidence is generated. | Versioned raw artifact, manifest, and reproducible command. |

No row may close by moving it to a later milestone while leaving its current claim or unsafe behavior intact. A feature may move to V3.8 only when V3.7 removes or narrows the corresponding public contract.

## Alternatives considered

### One unstructured V3.7 branch

This minimizes ceremony but combines migrations, ingestion, retrieval, security, Optimize, adapters, benchmarks, and packaging without isolation. Root-cause analysis and rollback would be poor. Rejected.

### Public 3.6.24 followed by V3.7

This provides the safest external recovery checkpoint but divides the market story and conflicts with the approved full-release strategy. Rejected as public sequencing; its internal separation principles are retained.

### V4 rewrite

This produces a cleaner blank sheet at the cost of losing tested behavior, delaying revenue, and introducing a larger migration. Rejected.

### One public V3.7 with internal gates

This preserves one market event while keeping engineering changes independently testable and reversible. Selected.

## Non-negotiable scope decisions

| Area | V3.7 decision | Why |
|---|---|---|
| Multi-machine memory federation | Not a V3.7 capability. Mesh is narrowed to authenticated coordination and verified cross-client continuity. | A real distributed memory needs membership, ordering, conflict resolution, replay, delete propagation, partitions, and recovery. Pretending one peer URL is federation is worse than a smaller truthful feature. |
| Retrieval channel count | Five independent candidate families maximum for V3.7: dense, lexical, graph, temporal, associative. | Correct independence and traceability matter more than restoring a seven-channel headline. |
| Quantized search tier | Remove the dead tier and claims unless an implementation plus benchmark is approved before code freeze. | Shipping an unreachable API is maintenance debt, not capability. |
| Semantic cache | Exclude from the stable product contract unless real production wiring and false-hit gates pass. Exact cache remains stable. | A semantic false hit returns a plausible wrong answer; safe absence is better than uncalibrated reuse. |
| Aggressive compression | Experimental and explicitly lossy. Safe mode and exact recovery are the stable contracts. | Aggregate “60–95%” language hides content- and mode-specific loss. |
| Cozo/Lance | Derived/optional accelerators unless full parity is proven. Remove primary-backend and zero-slowdown implications. | Initialization is not hot-path use. |
| Telemetry | Local counters plus explicit opt-in aggregate diagnostics; never memory content. | Product learning cannot violate the local-first promise. |
| LoCoMo for V3.7 | Historical paper results remain versioned historical evidence. V3.7 makes no current LoCoMo score claim, and a LoCoMo rerun is not a release gate. | The owner cannot rerun LoCoMo for this release. Reusing an old score as current would be false; blocking all correctness work on an unavailable run would also be wrong. |
| Local benchmark target | No target percentage. Run every safe laptop-feasible retrieval, lifecycle, scale, cache, compression, injection, and cross-client suite against a frozen RC, with raw outputs and no regression against a frozen local baseline. | Score chasing encourages leakage and protocol manipulation; reproducible local evidence can still prove release behavior without impersonating LoCoMo. |

## Engineering patterns

### Branch by abstraction

External MCP/CLI/HTTP contracts remain stable while canonical ingestion, retrieval, identity, and rendering services replace split implementations behind interfaces. Feature flags are temporary migration controls and must be deleted or permanently documented before stable release.

### Expand–migrate–contract

Database changes first add new fields/tables without removing old readers. A resumable backfill writes the new representation. Dual-read comparison proves parity. Only after rollback and upgrade tests pass may old fields and paths be contracted.

### Golden contract tests

Every external client executes the same remember, recall, update, forget, restart, and upgrade contract. Tests compare observable state rather than internal function calls.

### Evidence gates

A change is not complete because code was merged. It closes only when its ledger proof artifacts exist and the verification matrix passes against built packages.

## Definition of verified closure

A defect moves through:

```text
open → reproduced → fix_in_review → code_complete → proof_pending → verified
```

`verified` requires:

1. The exact acceptance criteria in the YAML ledger pass.
2. Required proof artifacts exist at stable paths.
3. The test runs against built release-candidate artifacts where applicable.
4. At least one review lane did not author the implementation.
5. Public copy and machine-readable metadata agree with the resolved contract.

No percentage-complete status is allowed. A row is either verified or still blocks release.

## Discovered-gap protocol

Implementation is expected to expose additional gaps. They are handled as follows:

1. Stop the affected workstream before extending the implementation around an unexplained failure.
2. Reproduce the behavior in an isolated environment and classify it as runtime-proven, code-proven, artifact-proven, documentation-only, or missing evidence.
3. Run GitNexus impact analysis for affected symbols and record upstream callers and execution flows.
4. Add a new stable ledger ID, severity, closure mode, dependencies, acceptance criteria, and proof paths before writing the fix.
5. If the gap affects data correctness, security, privacy, scope isolation, injection, migration, recovery, package integrity, benchmark integrity, or a public claim, it blocks the relevant release gate.
6. Add a failing regression first, implement the smallest complete contract correction, and rerun the broader affected matrix.
7. Give verification to a lane that did not author the implementation.
8. Close the row only from `proof_pending` to `verified` after built-artifact or runtime proof exists.

A new architectural preference, speculative optimization, or unrelated feature is not automatically a defect. It enters V3.7 only when evidence shows it is required to satisfy a frozen release contract. A material product expansion requires a short ADR with alternatives, consequences, migration cost, and explicit approval.

There is no numerical cap on new ledger rows. There is also no “we will fix it later” exception for a known gap inside the V3.7 contract.

## Risk controls

| Risk | Control |
|---|---|
| Schema/backfill corrupts user memory | Expand–migrate–contract, backup before migration, resumable checkpoints, dry-run audit, rollback rehearsal. |
| Broad retrieval rewrite hides regression | Golden query corpus, per-channel traces, old/new shadow comparison, bounded feature flag. |
| Security fix breaks existing clients | Capability negotiation, explicit protocol version, migration messaging, client matrix. |
| Benchmark tuning contaminates production | Frozen dataset hash, no gold evidence/summaries/category labels, clean adapter, separate dev/holdout analysis. |
| Feature count grows during program | Ledger freeze; additions require severity, evidence, closure mode, dependencies, and explicit approval. |
| One model rubber-stamps another | Runtime reproduction for behavior claims; code review alone labeled as code confirmation. |
| Long branch becomes unreleasable | Small contract-focused commits, internal artifacts, daily merge from main, change-impact review, RC code freeze. |

## Execution authorization boundary

This master program and its ledger are planning artifacts. Production-code work requires a second explicit authorization after the ledger and verification matrix are reviewed. The first implementation action after that authorization is GitNexus impact analysis for WS1/WS2 symbols, followed by failing isolation and daemon-identity contract tests. No ingestion rewrite begins before that gate is green.
